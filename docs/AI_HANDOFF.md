# AI Handoff

最近同步：`2026-04-25`  
状态：`active`

## 1. 这份文档的用途

这份文档是后续 AI 或人工接手当前项目时的快速入口。

它重点回答四个问题：

1. 这个项目现在是什么
2. 当前最重要的稳定结论是什么
3. 哪些核心约束绝对不能被回退
4. 下一轮最安全的接手路径是什么

## 2. 项目一句话

`Raccourci Control` 是一个基于 AutoHotkey v2 的 Windows 效率工具，核心同时包含：

- 输入提效器：全局热键、悬浮面板、字段/提示词/快捷字段
- 扩展容器：通过统一 mode 承载笔记、截图发手机、截图问答、简历自动填写、笔记显示等能力

## 3. 当前产品核心

- 全局热键可呼出

## 5.1 强制回退机制

只要用户明确提出“回退 / 撤回 / 恢复到之前版本 / 回到某个固定版本”，必须优先按下面顺序执行：

1. 优先回退到本次开发中已经明确保存的固定版本
2. 如果没有可用的 Codex 固定版本，则回退到上一个 git 提交版本
3. 如果用户明确指定某个 git 提交、标签、分支或文档锚点，则以用户指定目标为准

执行回退时，必须遵守以下规则：

- 先说明将要回退到哪个版本点，再执行
- 如果用户要求的是“整个版本回退”，默认按 git 整体回退处理，而不是只做局部删改
- 如果用户要求的是“只回退某一条需求”，才允许做定点移除
- 回退完成后，必须确认当前版本点、工作区状态，并默认重新启动 `main.ahk`
- 后续再次涉及回退，不再临时改规则，统一以本节为准

## 5.2 改动计数与 Git 检查点机制

从 `2026-04-26` 起，新增一条长期有效规则：

1. 每完成 1 次明确改动，且对应测试通过后，连续计数 `+1`
2. 每累计满 `3` 次改动，无论大小，都必须立即创建 1 个 git 检查点
3. 本项目里的 `git` 默认指：本地 commit 后继续直推 `origin/main`
4. 如果 `origin/main` 被分支保护或权限策略拒绝，才允许退回到远端工作分支
5. 只有 push 成功后，这次 git 检查点才算完成
6. 创建 git 检查点后，连续计数清零并重新开始
7. 高风险改动允许提前 git，不必硬等到第 3 次
8. 第一阶段开始前必须 git、第二阶段开始前也必须 git，这条旧规则继续保留
9. 连续 3 次改动期间，必须把每一次“具体改了什么”都写进流水文档，不能只留下计数
10. 流水文档只保留当前这一轮最多 `3` 次改动；更早历史统一通过 git 提交记录追溯
11. 每次改动只有在对应测试执行并确认通过后，才算这一次真正完成

配套文档如下：

- 规则文档：`docs/CHANGE_CHECKPOINT_RULE.md`
- 流水文档：`docs/CHANGE_ACTIVITY_LOG.md`

执行要求：

- 每次改动后，都要在 `docs/CHANGE_ACTIVITY_LOG.md` 记录当前轮次内容
- 每次改动都必须做对应测试；测试未做完或未通过时，这次改动不能算完成
- 每次 git 检查点后，都要更新 `docs/CHANGE_CHECKPOINT_RULE.md` 中的当前状态
- 每次 git 检查点说明时，都要明确说明是否已经 push 到远端，以及实际推送到哪个远端目标
- 每次流水记录至少写清楚“改了什么”；不要只写“已修改”这类空描述
- 每次流水记录还要写清楚“测试了什么、是否通过”
- 当第 3 次改动触发 git 并推送成功后，旧轮次流水不再继续长期保存在文档中
- 后续如用户再次询问“现在改了几次、什么时候该 git”，统一以这两份文档为准

## 6. 当前稳定共识

- 所有模块都必须先定义“核心主功能”，再做体验优化
- 任何次要问题都不能通过牺牲核心主功能来换稳定
- 面向外部环境的不确定性，一律按“通用兼容”处理，不能依赖定向兼容
- 新增任何开关、下拉、热键、配置项，必须同步完成四层：
  - 前端显示与回填
  - 后端读取与保存
  - 本地默认值与持久化
  - 运行时真正生效

## 7. 当前模块状态摘要

### 7.1 截图问答助手

当前核心定义：

- 非截图状态下，本地用户可见
- 截图结果中不可见
- 录屏结果中不可见
- 点击悬浮窗时，不应触发目标窗口 `window.blur`

当前已稳定接入的能力：

- 悬浮问答窗口
- `F1` 截图并问答
- 外部截图保护转发
- 录屏防捕捉基础链路
- 增强模式入口
- 四档透明度配置入口
- 测试页中的 API 基线测试

当前仍需谨慎处理的点：

- 任何对悬浮窗渲染、显示、恢复、透明度、保护态的修改，都有高回归风险
- 修体验时，必须先确认不会破坏“本地可见 / 截图不可见 / 录屏不可见”

恢复锚点口径：

- 如果截图问答链路再出问题，优先回到“核心保护链路不退化”的版本，再谈样式和交互优化

### 7.2 简历自动填写

当前核心定义：

- 本地资料可维护
- 插件可稳定读取并用于填充

当前稳定口径：

- 前端只维护字段名和值
- 别名与类型不再暴露为主 UI 编辑重点
- 后端字段表继续承担映射逻辑

### 7.3 笔记显示

当前核心定义：

- 主界面可维护 Markdown 内容
- 内容自动保存
- 自动解析目录
- 悬浮窗可读取自身笔记内容并显示

当前稳定口径：

- 悬浮窗已具备独立目录和正文区
- 已补目录上下快捷键
- 已补位置记忆
- 已做白窗异常的稳定化处理

当前注意点：

- 后续若继续改 notes overlay，优先保护生命周期、位置持久化、手动关闭锁定，不要先动视觉细节

## 9. 已知风险

- AHK 开发态存在自动热重载，调试时必须考虑 reload 带来的状态重建
- 多数能力横跨 AHK、PowerShell、Web 前端，单点修改极易造成契约断裂
- `AI_HANDOFF.md` 过去有编码与堆叠历史，当前版本已重写，后续应保持简洁，不再堆积过多实现细节
- 已建立临时故障档：`docs/incidents/TEMP_2026-04-26_storage_write_failure.md`；凡是“界面看似改了，但刷新/重启后未写入持久化文件”的问题，都应先按存储链路故障处理
