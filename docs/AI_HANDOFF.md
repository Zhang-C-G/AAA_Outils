# AI 交接文档（AI Handoff）

最近同步：`2026-04-24`  
状态：`active`

## 1. 这份文档的用途

这份文档用于在频繁切换 AI 协作者时，给下一位 AI 一个稳定、统一、低成本的接手入口。

目标不是替代模块文档，而是回答下面 5 个问题：

1. 这个项目整体是什么。
2. 当前已经做到哪一步。
3. 现在最重要的事情是什么。
4. 哪些约束不能被新一轮改动破坏。
5. 下一位 AI 接手后应该先读什么、先看哪里。

## 2. 项目一句话

`ZCG-Raccourci Control` 是一个基于 AutoHotkey v2 的 Windows 全局效率工具，核心同时包含：

- 输入提效器：通过全局热键、悬浮面板、快捷字段实现快速输入
- 扩展容器：通过统一模式切换承载笔记、截图发手机、截图问答、简历自动填写等能力

## 3. 当前稳定入口

- 运行入口：`main.ahk`
- 全局悬浮面板：`Alt+Q`
- 主配置界面：`Alt+Shift+Q`
- 截图问答悬浮窗：`Alt+Shift+A`
- 截图并问答：`F1`

## 6. 当前稳定共识

- 产品定位必须保持“双核心”：输入提效器 + 扩展容器。
- 模式切换入口必须保留，不要把扩展能力做成零散孤立功能。
- Web 配置与本地 AHK 状态必须保持契约一致；涉及保存、读取、模式、热键、Assistant 设置时，要同时检查前后端。
- 每次重要改动后必须同步更新文档，不能只改代码不改文档。
- 所有模块都必须执行“核心优先”规则：先保核心主功能，再处理附加体验；不能为了修次要 bug 牺牲主功能。

## 7. 当前特别需要新 AI 注意的点

- AHK 开发态会自动热重载；保存 `src/*.ahk` 后，脚本会自动 `Reload()`，调试时要显式考虑这一点。
- 涉及编码、配置写盘、热键映射、Assistant 安全链路的改动都属于高风险区域。
- Assistant 模块最近变化很多，当前稳定口径是：
  - 核心主功能：非截图状态下本地用户能持续看到问答悬浮窗；真实截图时必须先隐藏/隔离；录屏/共享时必须做到“本地可见、录屏里不见”
  - 硬约束：悬浮窗一经调用，就必须立即处于不可被录屏/共享捕捉的保护态；不能先短暂暴露，再补做保护
  - 硬约束：用户点击悬浮窗时，底层前台目标窗口不能失焦，不能触发目标应用自身的 `window.blur` / focus loss；悬浮窗交互必须尽量保持对目标窗口“不可检测”
  - 硬约束：只要检测到截图行为，悬浮窗必须确保不出现在截图结果中；不能被系统截图、区域截图或常见外部截图链路截入画面
  - 录屏防捕捉必须走通用方案，不能依赖某个录屏/会议软件的定向兼容
  - Web 页采用“基础设置 / 高级设置”分层
  - 高级设置默认隐藏，关闭时整块不显示
  - 顶部先展示快捷键说明
  - 模板按钮放回基础设置
  - `API Endpoint` 已从主 Web UI 移除
  - 截图保存目录可直接在主界面修改
  - “最新截图文件”已从主界面移除
  - 透明度改为四档：`20 / 50 / 75 / 100`
  - 新增“增强模式”开关，入口仅在主界面高级设置；默认关闭。增强模式本身也有独立核心，开启后必须围绕“非截图时本地可见、截图不可见、录屏不可见”执行，且不能与普通模式互相干扰；增强模式一旦开启就进入持续保护态，一旦保护状态失效且处于真正高风险场景时才允许保护性隐藏
- Assistant 悬浮窗当前稳定口径是：
  - 核心主功能优先级高于透明度、动效、样式等次级体验
  - 悬浮窗一旦显示，必须默认已进入防捕捉保护态，不能出现“先显示在录屏里、再隐藏”的瞬间泄露
  - 用户点击悬浮窗时，不能让底层目标窗口触发 `window.blur`、失焦或焦点切换；否则会暴露悬浮辅助行为
  - 只要检测到截图动作，悬浮窗必须先完成隔离或临时隐藏，保证最终截图结果里没有悬浮答案；这条规则对 `F1` 和外部截图快捷键一视同仁
  - 录屏期间本地必须可见
  - 截图时才允许临时隐藏
  - 回答区不再显示“回答区”字样
  - 透明度只允许在主界面修改，悬浮窗内不提供透明度入口
  - 不进入任务栏与常规 Alt-Tab 列表
  - 回答文本从左上角开始排版，并继续做自动换行与代码块整理
- Resume 模块当前稳定口径是：
  - 核心主功能：稳定维护字段名和值，并为浏览器插件提供可用的本地资料映射
  - 前端只维护字段名和值
  - 分区说明小字已从主 UI 移除
  - `aliases` 与 `type` 不再放在主 UI 上编辑
  - 后端默认字段表继续维护别名和类型

## 8. 当前优先事项

1. 保持文档与实现继续同步，避免多轮改动后口径漂移。
2. 截图问答模块后续若继续调 UI，优先保持“录屏本地可见 / 截图短隐藏 / WDA 隔离”的安全基线不被回退。
3. 简历自动填写模块后续若继续扩展，优先走后端字段表与插件规则，不要重新把别名和类型复杂度塞回主 UI。

## 8.1 Assistant 当前恢复锚点

- 当前主线不是继续堆 UI，而是守住截图问答助手的核心链路：本地可见、截图不可见、录屏不可见。
- 普通模式与增强模式都必须被视为独立核心模式；二者可以共享基础设施，但不能互相污染行为。
- 当前可作为恢复锚点的已知实现包括：
  - 主界面高级设置中的增强模式开关
  - 主界面四档透明度 `20 / 50 / 75 / 100`
  - 悬浮窗 NoActivate / `window.blur` 防暴露交互
  - `F1` 与悬浮窗按钮共用截图安全链路
  - 检测到截图行为时的临时隐藏 / 恢复机制
- 当前仍需持续验证的点包括：
  - 增强模式下悬浮窗打开与保护重挂接稳定性
  - 环境切换、切屏、display affinity 漂移时是否出现闪烁或自隐藏
  - “已保护但本地仍可见”的通用兼容性是否被后续改动破坏
- 当前已知剩余问题：
  - 点击悬浮窗内 `截图问答` 按钮后，悬浮窗存在自动隐藏的现象；该问题不影响当前“录屏不可见 / 截图不可见”的核心已实现版本，但属于下一步优先修复项
- 后续提交说明必须至少包含三段：
  - 这次实现了什么
  - 这次没有实现什么
  - 当前从哪里继续恢复最安全

## 9. 已知风险

- 文档量较大，如果不维护交接入口，后续 AI 很容易只看到局部。
- 一些功能同时横跨 AHK、PowerShell、Web 前端，单点修改容易造成契约偏差。
- AHK 热重载会让“代码改了马上生效”和“脚本重载导致状态重建”混在一起，需要注意排障判断。

## 10. 新 AI 接手时建议阅读顺序

1. `README.md`
2. `docs/AI_HANDOFF.md`
3. `docs/COMPONENTS.md`
4. 当前要改动对应的 `docs/modules/*.md`
5. 如涉及共享能力，再看 `docs/shared/*.md`
6. 如涉及接口 / 架构，再看 `docs/components/*.md`
