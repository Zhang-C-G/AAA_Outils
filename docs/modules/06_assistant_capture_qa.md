# 模块 06：截图问答助手

## 模块目标

- 通过快捷键或悬浮窗按钮触发截图并调用模型回答。
- 支持先启动悬浮窗，再按截图热键触发问答。
- 回答统一在本地悬浮窗展示，Web 端只负责配置与触发。
- 悬浮窗支持链路状态展示、长文本换行、代码段整理、滚动查看。
- 录屏期间悬浮窗对本地用户保持可见；截图时才允许短暂隐藏。
- 支持模板、多模板管理、模型下拉、限流、密钥保护与禁止复制。
- Assistant Web 页采用“基础设置 / 高级设置”分层，高级设置默认隐藏。

## 核心主功能

- 核心是：问答悬浮窗在非截图状态下对本地用户持续可见；真实截图时必须先隐藏/隔离；录屏/共享时必须做到“本地可见、录屏里不见”。
- 硬约束：悬浮窗一经调用，就必须立即处于不可被录屏/共享捕捉的保护态；不能先显示出来再补做保护。
- 硬约束：用户点击悬浮窗时，底层前台目标窗口不能失焦，不能触发目标应用自身的 `window.blur` / focus loss；悬浮窗交互必须尽量保持对目标窗口“不可检测”。
- 硬约束：只要检测到截图行为，悬浮窗必须确保不出现在截图结果中；不能被系统截图、区域截图或常见外部截图链路截入画面。
- 任何透明度、样式、布局、文案、按钮位置等次级问题，都不能以牺牲上述核心目标为代价。
- 录屏防捕捉必须采用通用方案，不能依赖对某个录屏软件、会议软件或共享工具做定向兼容。

## 增强模式

- 增强模式开关只放在主界面高级设置，不放在悬浮窗内。
- 默认关闭；开启后其目标描述必须是：非截图时本地可见、截图不可见、录屏不可见。
- 增强模式本身也有独立核心主功能：开启后就必须围绕“非截图时本地可见、截图不可见、录屏不可见”执行。
- 增强模式一旦开启，就必须进入持续保护态；不能只在截图瞬间才临时加强保护。
- 增强模式不得破坏录屏不见、截图不见等核心能力。
- 普通模式与增强模式必须是两套隔离逻辑，不能相互干扰；关闭增强模式后，系统必须完整回到普通模式行为。
- 运行时只有在防捕捉保护已生效时，增强模式才允许非截图阶段持续本地可见；截图链路本身仍必须优先隐藏/隔离。
- 增强模式下，只要保护状态发生漂移、切屏、环境切换或短暂失效，悬浮窗必须立即进入保护性隐藏；只有在保护状态重新确认有效后才允许恢复显示。

## 当前实现边界

- 文档中的“已实现”指当前代码已经接入对应链路或 UI，不等于在所有环境下都完成稳定性验证。
- 文档中的“核心已实现”表示功能链路已经存在，后续修复必须在不回退核心能力的前提下进行。
- 文档中的“增强模式”当前以持续保护为目标；若现场表现与目标不一致，应按 bug 处理，不能反向修改核心定义来迁就实现。
- 任何一次排障，如果修复了截图安全却回退了录屏不可见，或修复了录屏不可见却回退了本地可见，都视为不可接受回归。
- Assistant 新增配置项时，不能只新增 Web 开关或下拉。必须同步完成：
- 前端默认值与刷新回填。
- 后端 settings 默认值、读取、保存。
- 本地 AHK 配置读写。
- 运行时实际判断与生效逻辑。
- 这是一条必须反复执行的防错规则：如果新增了开关但没有写入后端/本地配置，刷新后状态丢失，一律视为未完成实现。

## 最终实现逻辑

- 非截图状态：
  - 悬浮窗默认持续显示给本地用户。
  - 悬浮窗显示前先建立防捕捉保护，避免先暴露再补保护。
  - 用户点击悬浮窗时不抢焦点，不触发底层 `window.blur`。
- 录屏 / 共享状态：
  - 悬浮窗继续对本地用户可见。
  - 依靠 `WDA_EXCLUDEFROMCAPTURE` 让录屏结果尽量看不到悬浮窗。
  - 若保护状态漂移，系统优先重挂保护，而不是直接关闭窗口。
- 外部截图状态：
  - `PrintScreen` / `Win+Shift+S` / `Ctrl+Alt+A` 一律先临时隐藏悬浮窗，再转发截图动作。
  - 目标是优先保证任何外部截图结果中都没有悬浮窗。
- `F1` 内部截图状态：
  - 普通模式：沿用安全基线，截图前先短隐藏，再截图并恢复。
  - 增强模式：先强制重挂一次防捕捉保护。
  - 若重挂成功：直接执行内部截图，保持本地可见，同时让截图结果不包含悬浮窗。
  - 若重挂失败：自动回退到短隐藏截图，确保结果安全优先。
- 问答结果更新：
  - 悬浮窗实例保持不变。
  - 截图完成后继续在同一窗口中显示“分析中”状态。
  - 模型返回后直接在同一悬浮窗内更新答案，不重建新窗。

## 悬浮窗行为

- 悬浮窗仅保留标题、状态栏、`截图问答` 按钮与回答展示区。
- 悬浮窗不进入任务栏与常规 Alt-Tab 列表。
- 回答区禁止复制、禁止选中、鼠标光标固定为箭头。
- 悬浮窗以 NoActivate 模式显示，点击不抢焦点。
- 回答区允许鼠标滚轮滚动长答案，但仍禁止鼠标选中与复制，不放开文本聚焦。
- 回答区不再显示“回答区”提示字样；仅在内容超长时显示当前行号范围。
- 回答文本从左上角起始渲染，不再垂直居中。
- 回答文本会做可读性整理：
  - 自动换行
  - 代码块 fence 整理
  - 多余空白压缩
  - 中文句读后的弱换行优化
- 展示实现已从 `Edit` 子控件收敛为轻量只读文本渲染，保留 `Alt+Up / Alt+Down` 滚动。
- 首次显示会先完成本地重绘，再延后挂接捕捉保护，降低按钮区域黑块与透明度切换闪烁。

## 录屏 / 截图安全策略

- 录屏 / 会议 / 共享期间：
  - 悬浮窗对本地用户必须持续可见
  - 录屏参与者看到的共享画面中不应出现悬浮窗内容
  - 悬浮窗一旦被调用，默认就要已经处于防捕捉状态，不能先暴露在录屏结果里再切到保护态
  - 用户点击悬浮窗时，不能让底层目标窗口触发 `window.blur`、失焦或焦点切换
  - 不能因为检测到录屏进程或窗口就自动隐藏
  - 目标是“本地能看见，录屏里看不见”
  - 防捕捉方案必须对随机录屏软件尽量成立，不能把实现建立在 OBS / Zoom / 腾讯会议 / Teams / 飞书 等某个软件的专用兼容逻辑上
- 截图期间：
  - 只要检测到截图动作，悬浮窗必须立即先完成隔离或临时隐藏，确保最终截图结果里没有悬浮窗内容
  - `PrintScreen` / `Win+Shift+S` / `Ctrl+Alt+A` 命中时先临时隐藏悬浮窗，再转发截图动作
  - `F1` 内部截图在普通模式下先隐藏再截图；在增强模式下优先尝试“本地可见截图”，仅在保护重挂失败时回退到隐藏截图
- 防捕捉基线：
  - 第一层：`WDA_EXCLUDEFROMCAPTURE`
  - 第二层：真实截图动作触发时临时隐藏 + 原位恢复
- 保护稳定性：
  - 先读 `GetWindowDisplayAffinity`
  - 仅在漂移或超时场景按需重建 affinity
  - 关键日志：`assistant_overlay_protect_rearm`、`affinity_drift_*`

## 2026-04-24 新增业务：按住语音输入

- 新业务定位：这是截图问答助手上的新增输入链路，不替代截图问答主链。
- 解耦原则：
- `F1` 仍然只负责截图并问答。
- 新增 `F3` 默认热键，按住时进入语音监听，松开时自动结束监听并发起文本问答。
- 该链路不应影响“本地可见、截图不可见、录屏里不可见”的截图问答核心保护目标。

- 当前实现口径：
- 已新增快捷键：`assistant_voice_input`，默认值 `F3`，支持进入统一快捷键配置体系。
- 已新增语音 provider 接口字段：
- `voice_input_provider`
- `voice_input_endpoint`
- `voice_input_model`
- 当前默认 provider 为 `local_windows`，即本地 Windows 语音识别。
- 当前保留 `mock_local` 作为本地模拟 provider，用于后续调试 / 扩展。

- 当前业务流：
- 按住 `F3`：悬浮窗显示“正在监听输入”，并启动本地语音识别会话。
- 松开 `F3`：停止语音识别，读取识别文本。
- 若识别文本为空：悬浮窗提示“未识别到语音内容”。
- 若识别成功：将识别文本作为文本问题发送给当前问答模型，并在悬浮窗中展示“语音输入 + 回答”。

- 当前说明：
- 该业务属于新增能力，已按独立链路接入。
- 截图主链、录屏防捕捉主链、外部截图保护主链，本轮不应因语音输入能力而退化。
