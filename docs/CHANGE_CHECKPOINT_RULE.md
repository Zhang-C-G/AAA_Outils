# 改动计数与 Git 检查点规则

最近同步：`2026-04-26`
状态：`active`

## 1. 目的

这份文档只负责一件事：

- 记录当前连续改动次数
- 规定何时必须创建 git 检查点
- 给后续回退、交接、补档提供统一锚点

## 2. 强制规则

从现在开始，执行以下规则：

1. 每完成 1 次明确改动，且对应测试通过后，计数 `+1`
2. 每累计满 `3` 次改动，无论大小，必须立即创建 1 个 git 检查点
3. 本项目语境里的 `git`，默认不是只做本地 commit，而是：
   - 本地 `commit`
   - 默认直接推送到 `origin/main`
4. 如果 `origin/main` 因分支保护、权限或远端策略拒绝直推，才允许退回到“推送到当前工作分支对应的远端分支”
5. 只有当远端 push 成功后，这次 git 检查点才算真正完成
6. 创建 git 检查点并成功 push 后，连续改动计数清零
7. 如遇高风险改动，可在未满 3 次前提前 git
8. 进入第一阶段前，仍然必须先 git
9. 进入第二阶段前，仍然必须再 git
10. 涉及用户明确要求的回退时，仍以 `docs/AI_HANDOFF.md` 中的回退机制为准
11. `docs/CHANGE_ACTIVITY_LOG.md` 只保留“当前连续这一轮”的最多 `3` 次改动，不继续长期累积旧历史
12. 当累计到第 3 次并执行 git 前，必须确保这连续 3 次各自改动内容都已经写入流水文档
13. 第 3 次改动完成并成功 git 后，旧一轮的长期追溯统一交给 git，本流水文档进入下一轮重新记录
14. 任何改动只有在对应测试执行且通过后，才算“完成”；未测试或测试失败都不能计入这 3 次改动

## 3. 什么算 1 次改动

按“一个连续目标的落地”计数，不按改了多少行、多少文件计数。

### 计为 1 次的例子

- 修复 1 个启动报错
- 给 1 个模块补 1 组 UI 字段
- 新增 1 份阶段文档
- 修复 1 条明确链路

### 不这样计数

- 不是改 3 个文件就算 3 次
- 不是改 20 行就算 20 次
- 不是开了 3 个终端命令就算 3 次

## 5. 维护方式

每次 Codex 或人工改动后，都要同步两件事：

1. 在 `docs/CHANGE_ACTIVITY_LOG.md` 维护当前轮次的 1 条流水
2. 更新本文件中的“当前连续改动次数”“上一个 git 检查点”等状态

补充要求：

- `docs/CHANGE_ACTIVITY_LOG.md` 里的每一条记录，至少要写清楚“这次改了什么”
- `docs/CHANGE_ACTIVITY_LOG.md` 里的每一条记录，还要写清楚“做了什么测试、测试是否通过”
- 本流水文档最多只保留当前轮次的 `3` 次改动
- 到第 `3` 次触发 git 并推送成功后，旧轮次历史不再继续留在文档里，统一通过 git 追溯
- 因此每一轮的第 `1-3` 次记录都要写得足够清楚，保证回退到上一个 git 前后都知道这轮在做什么

## 6. git 提交格式建议

建议 git 检查点提交信息统一带上 `checkpoint` 或 `archive` 关键词，例如：

- `checkpoint: assistant stage1 ui round 1`
- `archive: before resume stage2 wiring`
- `checkpoint: after 3 incremental changes`

## 7. 远端要求

- 默认远端：`origin`
- 默认目标：`origin/main`
- 仅当 `origin/main` 无法直推时，才允许退回当前工作分支对应的远端分支
- 如果 push 失败，不能口头视为“已 git”
- 必须在结果说明里明确：
  - commit hash
  - 当前分支名
  - 实际推送目标
  - push 是否成功
