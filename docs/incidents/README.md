# Bug / Incident 文档机制

最近同步：`2026-04-26`

## 1. 这份文档解决什么问题

这份文档专门约束 `docs/incidents/` 目录怎么用，避免后面再出现：

- 出了 bug 但没有建档
- 建了 TEMP 但越写越散
- 修好了却没有归档
- 同类问题反复出现，但没有留下可复用结论

## 2. 什么时候必须建 bug 文档

出现下面任意一种情况，就建：

- 主链路真实故障
- 刷新 / 重启 / 重进后状态不一致
- 保存成功但落盘结果不一致
- AHK / Web / 本地文件三层状态不一致
- 已经连续两轮以上还没打中根因的问题

## 3. 放在哪里

统一放在：

- `docs/incidents/`

## 4. 文件命名规则

处理中：

- `TEMP_YYYY-MM-DD_xxx.md`

确认修复并归档后：

- `RESOLVED_YYYY-MM-DD_xxx.md`

命名要求：

- 用日期
- 用短横线或下划线表达主题
- 标题必须能看出故障核心，不要只写 `bug1`、`test`、`fix`

## 5. TEMP 文档必须写什么

TEMP 文档至少写清楚这 6 件事：

1. 现象
2. 影响范围
3. 复现步骤
4. 当前结论
5. 最可疑根因
6. 下一步怎么验证

重点不是“写得漂亮”，而是让下一轮接手时能继续查，不用重新猜。

## 6. RESOLVED 文档必须写什么

RESOLVED 文档至少写清楚这 5 件事：

1. 当时的问题是什么
2. 最终根因是什么
3. 修了哪些文件
4. 做了哪些测试
5. 后续如何避免同类问题

## 7. 标准处理流程

1. 先建 `TEMP_*.md`
2. 在文档里锁定现象、范围、复现步骤
3. 修代码
4. 做真实测试
5. 如果没彻底解决，继续保留 TEMP 并更新当前结论
6. 如果确认解决，转成 `RESOLVED_*.md`
7. 补 `ACTION_LOG.md`
8. 必要时补 `AI_HANDOFF.md`

## 8. 和其他文档的关系

- 模块说明文档：讲模块本身该怎么工作
- `docs/incidents/`：讲“这次为什么坏了、怎么修的”
- `AI_HANDOFF.md`：只保留当前最关键的交接风险
- `ACTION_LOG.md`：记录发生了什么变化

不要把 incident 文档写成模块说明文档，也不要把模块说明文档写成故障流水账。

## 9. 推荐模板

统一使用：

- `docs/templates/INCIDENT_TEMPLATE.md`

## 10. 当前约束

以后凡是涉及：

- 保存失败
- 字段不加载
- 栏目重进后消失
- 热重载后状态错乱
- AHK / Web / `config.ini` 三层不一致

一律优先更新或新建对应 incident 文档，再继续改代码。
