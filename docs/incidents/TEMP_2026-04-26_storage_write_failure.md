# TEMP: A 模块或其他模块信息未成功写入存储层

日期：`2026-04-26`
状态：`排查中，但已补上阶段性保护`

## 1. 现象

- A 模块栏目或字段在界面里看起来已经改完
- 但刷新、重新进入、或后续再次调用后，改动可能消失
- 表现上像“自动保存没生效”或“没有真正写入后端”

## 2. 当前结论

当前已经确认两件事：

1. Web 后端 `/api/save` 到 `config.ini` 的真实写盘链路是通的
2. 问题更像是“Web 刚写进去，后面又被 AHK 旧内存覆盖回去了”

也就是说，问题不只是在前端触发保存，还涉及保存后的状态同步与后续覆盖风险。

## 3. 已确认的链路事实

- `/api/save` 可以把测试栏目和字段真实写入 `config.ini`
- `/api/state` 可以把这些刚写入的数据正常读回来
- 空栏目本身也可以被后端正常写入并回读
- A 模块栏目新增、改名、删除、拖拽排序已经改成立即保存

## 4. 阶段性根因判断

当前最可疑的根因是：

- AHK 侧原本主要依赖 `action file` 感知“Web 刚保存过，需要 reload”
- 如果这个 `reload` 时序被错过，AHK 后续某次 `SaveData()` 仍可能拿旧的 `gCategories / gData` 整文件重写
- 这样就会出现：
  - Web 保存看起来成功
  - 但后面重新进入时，栏目或字段又丢了

## 5. 本轮已做修复

### 修复 1：结构改动立即保存

以下 4 类结构改动已经不再依赖延迟自动保存，而是立即落盘：

- 新增栏目
- 栏目改名
- 删除栏目
- 栏目拖拽排序

### 修复 2：首屏预取

当上次退出停留在 A 模块时，启动首屏直接预取 A 模块数据，减少重新进入时的等待。

### 修复 3：AHK SaveData 磁盘新版本保护

在 `SaveData()` 前新增了第二道保护：

- 先继续处理已有的 web reload action
- 再比较当前 `config.ini` 的磁盘修改时间
- 如果磁盘版本比 AHK 内存新，先 `ReloadAppStateFromDisk()`
- 然后才继续保存

这一步的目的就是避免“旧内存覆盖刚写入的新数据”。
