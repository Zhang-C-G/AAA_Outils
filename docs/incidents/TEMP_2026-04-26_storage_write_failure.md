# TEMP: A 模块或其他模块信息未成功写入存储层（2026-04-26）

## 现象

- A 模块或其他模块在界面上看起来已经修改完成，但刷新、重启或重新打开后，信息没有成功保留下来。
- 表现可能包括：
  - 新增栏目没有真正写入
  - 改名结果刷新后回退
  - 字段内容修改后丢失
  - 其他模块配置项显示已更新，但对应持久化文件没有变化
- 当前语境中的“数据库”主要指项目实际维护的持久化载体，例如：
  - `config.ini`
  - 模块对应的本地配置文件
  - Web 配置服务负责读写的状态文件

## 根因

- 根因暂未最终确认，当前按“持久化链路异常”处理。
- 需要重点排查以下方向：
  - 前端改动未真正触发保存
  - 自动保存触发了，但后端接口未成功落盘
  - 后端已返回成功，但写入目标文件失败或写入到错误位置
  - 模块加载的是旧数据源，导致看起来像“没保存”
  - 写入成功后又被后续流程覆盖回旧值
- 当前已确认的一条阶段性结论：
  - 后端 `/api/save` 到 `config.ini` 的真实写盘链路可工作，已通过临时测试栏目闭环验证。
  - 当前更可疑的是前端触发时机，尤其是 A 模块栏目改名这种依赖 `blur` + 自动保存延迟的链路。

## 修复

- 当前尚未完成正式修复，本文件用于锁定问题与后续排查范围。
- 后续修复时，至少要逐段确认：
  - 前端是否触发了真实保存动作
  - 后端接口是否收到正确 payload
  - 持久化目标文件是否实际发生变化
  - 刷新、重启、重新打开后是否仍保持一致
- 当前已先做一条收敛修正：
  - A 模块栏目改名在提交后改为立即触发保存，不再使用默认的延迟自动保存。
