# 文档创建指南

最近同步：`2026-04-26`

## 1. 这份指南解决什么问题

这份指南不是讲“代码怎么写”，而是讲“开发过程中，各种文档应该怎么建、什么时候建、写到什么粒度”。

它主要覆盖：

- 第一阶段文档怎么建
- 第二阶段能力说明文档怎么建
- 悬浮窗第二阶段专项文档怎么补
- 故障 / bug 文档怎么建
- `AI_HANDOFF.md` 怎么更新
- `ACTION_LOG.md`、`DOC_CHANGELOG.md` 怎么追加

## 2. 创建文档前的统一动作

在新建或大改文档前，先做这几步：

1. 先读 `docs/extension/AI_DEVELOPMENT_PLAYBOOK.md`
2. 如果要进入第一阶段或第二阶段，先做 git 检查点
3. 如果能力涉及悬浮窗，再读 `docs/extension/OVERLAY_STAGE2_GUIDE.md`
4. 找到目标模块现有文档：`docs/modules/*.md`
5. 判断这次是“新建文档”还是“在原文档上增加新章节”

默认原则：

- 能挂在原模块文档上的，优先挂原模块文档
- 只有当内容明显独立、会长期维护、或者是故障归档时，才新建独立文档

## 3. 第一阶段文档怎么建

### 3.1 什么时候建

在模块刚进入第一阶段时就要建，不要等界面做完再补。

### 3.2 建在哪里

优先放在对应模块文档里：

- `docs/modules/*.md`

如果当前模块还没有正式模块文档，再先建模块文档。

### 3.3 第一阶段文档必须写什么

第一阶段文档的目标，是把“壳层和结构”讲清楚。至少要写：

1. 模块目标
2. 核心主功能
3. 这阶段只做什么
4. 这阶段明确不做什么
5. 页面入口 / 模式入口
6. 页面布局与主要区块
7. 关键控件占位
8. 当前空态 / 默认态
9. 进入第二阶段后要补什么

### 3.4 第一阶段文档不要写成什么样

不要把它写成：

- 大而空的需求作文
- 只写“有按钮、有输入框”的表面说明
- 已经进入第二阶段才会确定的技术细节

第一阶段文档要回答的是：

`这个模块的壳子是什么样，边界是什么样，后面往哪里接。`

### 3.5 推荐模板

可直接参考：

- `docs/templates/STAGE1_DOC_TEMPLATE.md`

## 4. 第二阶段能力说明文档怎么建

### 4.1 什么时候建

不是做完再写，而是：

1. 第一阶段完成后
2. 第二阶段首次真实开发前

### 4.2 建在哪里

优先继续写在模块文档里，新增“第二阶段能力说明”或“能力说明”章节。

如果模块很复杂，也可以单独建一份模块能力说明文档，但要在模块主文档里明确链接过去。

### 4.3 第二阶段文档必须写什么

第二阶段文档必须按“用户行为”拆，不按代码文件名拆。

至少要写：

1. 用户会做什么动作
2. 每个动作触发后系统怎么响应
3. 数据从哪里读、往哪里存
4. 哪些状态要持久化
5. 刷新后会怎样
6. 重启后会怎样
7. 哪些行为允许，哪些行为禁止
8. 异常时会怎样
9. 验收时要看什么
10. 恢复锚点是什么

### 4.4 第二阶段文档最容易漏什么

高频漏项有：

- 只写保存，不写刷新恢复
- 只写页面内刷新，不写浏览器原生刷新
- 只写前端显示，不写后端与本地持久化
- 只写“能点”，不写失败路径
- 只写主路径，不写草稿恢复 / 自动保存

### 4.5 推荐模板

可直接参考：

- `docs/templates/STAGE2_CAPABILITY_TEMPLATE.md`

## 5. 悬浮窗第二阶段专项文档怎么补

如果能力带悬浮窗，仅有普通第二阶段文档还不够，还要额外对照：

- `docs/extension/OVERLAY_STAGE2_GUIDE.md`

此时文档里必须明确写清：

1. 显示 / 隐藏 / 关闭 / 恢复
2. 位置记忆
3. 焦点影响
4. 快捷键作用域
5. 保护态 / 防截图 / 防录屏
6. 恢复策略
7. 相关日志点

## 6. bug / 故障文档怎么建

### 6.1 什么时候建

只要出现真实故障、回归、链路断裂、线上异常，就建。

### 6.2 建在哪里

放在：

- `docs/incidents/`

### 6.3 命名规则

处理中：

- `TEMP_YYYY-MM-DD_xxx.md`

修复归档后：

- `RESOLVED_YYYY-MM-DD_xxx.md`

### 6.4 必须写什么

1. 现象
2. 根因
3. 修复
4. 影响文件
5. 验收结论

### 6.5 推荐模板

可直接参考：

- `docs/templates/INCIDENT_TEMPLATE.md`

## 7. AI_HANDOFF 怎么更新

`docs/AI_HANDOFF.md` 不是流水账，而是交接锚点。

当这些内容变化时必须更新：

1. 当前产品核心
2. 当前优先事项
3. 已知风险
4. 恢复锚点
5. 接手顺序

不要把它写成：

- 代码 diff 汇总
- 无重点的长日志
- 过多细碎实现细节

它应该回答的是：

`下一位接手的人，现在最该知道什么。`

## 8. ACTION_LOG 怎么写

`docs/ACTION_LOG.md` 记录的是“发生了什么变化”。

适合记录：

- 新增了什么能力
- 修复了什么故障
- 调整了什么交互口径
- 完成了什么阶段性收口

写法要求：

- 单行
- 先写日期
- 再写动作结论
- 不写长段推理

## 9. DOC_CHANGELOG 怎么写

`docs/DOC_CHANGELOG.md` 只记录文档层面的变化。

适合记录：

- 新增了哪些文档
- 重写了哪些旧文档
- 哪些文档与代码完成了同步

它不代替 `ACTION_LOG.md`，只负责文档自己的变更记录。

## 10. 推荐创建顺序

### 10.1 新模块

1. git 检查点
2. 第一阶段文档
3. 第一阶段实现
4. git 检查点
5. 第二阶段能力说明文档
6. 第二阶段实现
7. 验收
8. 更新 `AI_HANDOFF.md`
9. 更新 `ACTION_LOG.md`
10. 更新 `DOC_CHANGELOG.md`

### 10.2 旧模块扩展

1. 先判断是第一阶段扩结构，还是第二阶段接链路
2. 进入对应阶段前先做 git 检查点
3. 先补文档，再动代码
4. 验收后补日志与交接

### 10.3 故障修复

1. 建 `TEMP` 或直接建 `RESOLVED` 故障文档
2. 修复代码
3. 验证链路
4. 归档故障文档
5. 补 `ACTION_LOG.md`
6. 必要时补 git 检查点
