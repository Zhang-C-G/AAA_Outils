# 悬浮窗第二阶段开发指南

最近同步：`2026-04-25`

## 1. 适用范围

这份文档只用于悬浮窗类能力的第二阶段开发。

如本阶段开发过程中涉及“回退 / 撤回 / 恢复固定版本”，必须先遵循 `docs/AI_HANDOFF.md` 中的“5.1 强制回退机制”，再执行具体回退动作。

典型场景：

- 截图问答悬浮窗
- 笔记显示悬浮窗
- 后续任何需要常驻显示、快捷键呼出、具备保护态或特殊交互的悬浮窗

## 2. 为什么单独写

悬浮窗类能力的问题通常不出在界面，而出在第二阶段漏项。

历史高频漏项：

- 只做了显示，没做隐藏 / 关闭 / 恢复
- 只做了可见，没做位置保存
- 只做了按钮，没做热键作用域
- 只做了内容显示，没做重载 / 刷新 / 重启恢复
- 只修一个视觉问题，却破坏了核心保护链路

## 3. 第二阶段强制清单

### 3.1 生命周期

- 入口调用是否稳定
- 重复呼出是否复用同一实例
- 关闭是否彻底
- 隐藏后是否还能正确恢复
- 手动关闭后是否阻止错误自动恢复

### 3.2 位置与尺寸

- 是否有默认位置
- 是否保存最后位置
- 下次是否恢复到相同位置
- 越界时是否自动校正回屏幕范围

### 3.3 焦点与交互

- 是否会抢焦点
- 是否会影响底层目标窗口
- 是否会触发目标页面 `blur / focus loss`
- 鼠标滚轮、键盘、按钮行为是否相互独立

### 3.4 热键

- 是否定义独立作用域
- 是否只在悬浮窗可见时生效
- 是否与已有热键冲突
- 前后端和本地默认值是否同步

### 3.5 持久化

- 配置项是否写回
- 刷新后是否保留
- 重启后是否保留
- AHK 重载后是否仍正常

### 3.6 日志

至少要有：

- `open`
- `show`
- `hide`
- `close`
- `restore`
- `protect on/off`
- 异常失败

关键节点没有日志，后续排障成本会非常高。

## 4. 如果悬浮窗带保护态

必须额外检查：

1. 本地可见逻辑
2. 截图不可见逻辑
3. 录屏不可见逻辑
4. 保护失效后的兜底逻辑
5. 不允许为了修闪烁而直接取消核心保护

## 5. 推荐开发顺序

1. 先稳定显示 / 隐藏 / 关闭
2. 再补位置保存
3. 再补热键作用域
4. 再补保护态
5. 最后再调透明度、动效、视觉细节

## 6. 完成标准

一个悬浮窗的第二阶段，只有下面都成立才算完成：

- 可稳定呼出
- 可稳定关闭
- 可稳定再次打开
- 位置可记忆
- 配置不丢
- 热键不串
- 日志可排障
- 核心主功能未被次要修复破坏


## Git 检查点补充规则

- 悬浮窗类能力正式进入第二阶段前，必须先做 git 检查点。
- 原因：悬浮窗链路高风险，改挂后必须能快速回退。
