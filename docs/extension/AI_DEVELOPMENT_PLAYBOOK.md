# AI 开发总指南
最近同步：`2026-04-25`

## 1. 定位

这份文档是 AI 进入任何一个开发阶段前的强制入口。
它解决的不是“某一段代码怎么写”，而是：

- 先做什么
- 后做什么
- 哪些不能漏
- 哪些不能为了次级体验去破坏核心功能

## 2. 强制入口规则

- 每次开始一个新的开发阶段前，都必须先读取本文件。
- 这里的“开发阶段”包括：
  - 新模块第一阶段：界面与结构阶段
  - 新模块第二阶段：真实功能接线阶段
  - 旧模块新增配置项
  - 旧模块新增悬浮窗
  - 旧模块问题排障与回归修复
- 如果本轮涉及悬浮窗，还必须继续读取 `docs/extension/OVERLAY_STAGE2_GUIDE.md`。
- 如果本轮涉及具体业务模块，还必须继续读取对应的 `docs/modules/*.md`。
- 结论：本文件不是参考资料，而是每轮开发前的前置规范。
- 如果本轮进入第二阶段，还应继续读取 `docs/extension/全局私人偏好文档.md`，并按需进入 `docs/extension/global_preferences/` 下的主题子文档。

## 3. 开发总原则

### 3.1 核心优先

- 每个模块开工前，必须先写清楚“核心主功能是什么”。
- 后续所有视觉调整、体验优化、按钮增减、交互微调，都不能以牺牲核心主功能为代价。
- 如果目标冲突，先保核心，再保次级体验。

### 3.2 通用兼容优先

- 不允许把核心能力建立在某一个特定软件、特定网站、特定浏览器插件或某一个特定录屏工具的定向兼容上。
- 默认按“随机环境、随机目标、随机截图/录屏链路”设计。

### 3.3 四层同步

凡是新增开关、下拉、快捷键、按钮、文本配置、状态位，都必须同步检查四层：

1. 前端显示与回填
2. 后端读取与保存
3. 本地持久化默认值
4. 运行时真实生效

缺任意一层，都不算完成。

### 3.4 文档同步

- 改模块：更新 `docs/modules/*.md`
- 改共享契约：更新 `docs/shared/*.md`
- 改架构或接口：更新 `docs/components/*.md` 或相关系统说明文档
- 改阶段结论或恢复锚点：更新 `docs/AI_HANDOFF.md`

## 4. 所有模块统一采用两阶段开发法

这条规则不只适用于悬浮窗，而是适用于每一个模块。
可以把它理解为：先把“壳和入口”做稳，再把“真实能力”接齐。

### 4.1 第一阶段：模块界面与结构阶段

第一阶段的目标，是先把模块的结构、入口、布局、状态占位搭起来。
这一阶段重点是“看得见、进得去、结构稳定”，不是追求业务已经完全可用。

第一阶段必须完成：

1. 模块名称、定位、核心主功能说明
2. 模块入口、标签页、切换入口或导航入口
3. 页面布局、区块划分、标题层级
4. 必要的按钮、开关、下拉、输入区、说明区
5. 状态占位、空态文案、默认展示内容
6. 与现有模块的页面关系，不相互挤压、不明显破版
7. 模块文档先写清楚：目标、边界、后续第二阶段要补什么

第一阶段明确不要求全部完成的内容：

- 后端真实保存
- 真实接口接线
- 快捷键真实注册
- 悬浮窗保护链路
- 完整异常处理

第一阶段完成标准：

- 界面结构已经稳定
- 用户已经能看懂这个模块将来做什么
- 第二阶段不需要再推翻整体布局

### 4.2 第二阶段：模块真实功能开发阶段

第二阶段的目标，是补齐完整业务链路，让模块从“有壳”变成“可用”。
这一阶段要做的，不只是把按钮点通，而是把前端、后端、本地配置、运行时行为全部闭环。

第二阶段不是一上来直接写代码。
第二阶段的第 1 步，必须先写“能力说明文档”。

这份文档的作用是：

- 先定义这次要实现的能力到底包含什么
- 先定义默认交互，不允许边写边猜
- 先定义允许行为、禁止行为、持久化要求、异常处理要求
- 让后续实现和检查都以这份文档为准

这份文档的颗粒度必须足够细，默认按“用户实际操作行为”拆分，而不是按代码模块名拆分。

也就是说，描述单位优先是：

- 用户点击了什么
- 用户输入了什么
- 用户失焦时会怎样
- 用户按回车时会怎样
- 用户拖动后会怎样
- 用户刷新后应该看到什么

而不是只写：

- 有重命名
- 有删除
- 有拖动

后者太粗，无法支撑第二阶段的真实实现与验收。

在标准能力说明文档写完之后，还要继续对照：

1. `docs/extension/全局私人偏好文档.md`
2. `docs/extension/global_preferences/*.md` 中与本次能力相关的主题子文档

然后再融合成“最终能力说明文档”。

第二阶段必须完成：

1. 先写能力说明文档，并持续维护
2. 读取全局私人偏好总索引
3. 读取与本次功能相关的全局私人偏好子文档
4. 形成最终能力说明文档
5. 后端状态读取、保存、聚合返回
6. 本地默认值、持久化、重启恢复
7. 前端回填、保存、刷新后不丢
8. 运行时真实生效，而不是只改了界面
9. 快捷键注册、反注册、作用域、冲突检查
10. 异常处理、边界处理、失败兜底
11. 最低限度的自测与回归验证
12. 文档补齐“已实现 / 未实现 / 风险 / 恢复锚点”

第二阶段完成标准：

- 刷新后不丢
- 重启后不丢
- 运行时真实生效
- 与已有模块不互相破坏
- 后续排障能通过日志和文档继续追踪

### 4.4 通用验收方向

通用验收指南只提供方向，不代替每个模块自己的专项验收文档。
原因很简单：每个模块都有独特行为，专项模块仍然要维护自己的核心检查项。

但所有模块都必须额外检查下面这些通用项：

1. 页面内刷新是否正常
2. 浏览器原生刷新是否正常
3. 自动保存未完成时，是否有草稿兜底恢复
4. 后端真实落盘后，重新读取是否一致
5. 重启主程序后，状态是否仍一致
6. 错误时是否至少有日志或提示能定位
7. 如果模块存在“栏目 / 分区 / 标签”这类标题结构，标题是否支持直接改名

这里特别强调：

- “点击页面自己的刷新按钮”
- “按 F5 / Ctrl+R / 浏览器地址栏刷新”

这两类刷新不是同一条链路，必须分开验收。

历史上已经出现过的问题是：

- 页面内刷新做了草稿恢复
- 浏览器原生刷新没有走这条恢复链路
- 结果表现为“刚修改条目，刷新后丢失”

因此，以后凡是带自动保存的模块，都必须明确检查“原生刷新兜底”。

推荐做法：

- 对可编辑模块统一提供“会话草稿兜底”
- 页面内刷新、浏览器原生刷新、页面隐藏这三类离开动作都尽量复用同一套草稿写入逻辑
- 页面重新加载后优先尝试恢复草稿，再继续后续交互
- 对“栏目 / 分区 / 标签”这类结构，优先提供“点击标题直接改名”的统一交互，而不是把改名藏成次级操作

### 4.5 第二阶段触发时机

第二阶段检查和能力说明文档的首次触发时机，不是功能做完以后，而是：

- 第一阶段完成后
- 准备进入第二阶段首次真实开发时

此时就要先写能力说明文档，再开始实现。

更完整地说，第二阶段首次真实开发前的文档顺序应当是：

1. 先写标准能力说明文档
2. 再读取全局私人偏好总索引
3. 再读取与本次能力相关的子文档
4. 融合形成最终能力说明文档
5. 再按最终能力说明文档开始实现

如果第二阶段中新增了某种关键能力，也要立即补对应的能力说明，而不是等全部做完再回头补。

例如：

- 做了栏目结构，就要先写栏目能力说明
- 做了悬浮窗结构，就要先写悬浮窗能力说明
- 做了自动保存结构，就要先写自动保存能力说明

### 4.6 能力说明文档建议结构

能力说明文档建议至少包含：

1. 该能力的目标
2. 用户会进行哪些操作
3. 每种操作触发后的系统响应
4. 每种操作的边界情况
5. 哪些结果必须持久化
6. 哪些行为允许，哪些行为禁止
7. 默认交互方式是什么

如果做不到按这些点写清楚，说明这项能力在第二阶段里还没有被定义完整。

### 4.3 阶段切换规则

- 不允许第一阶段还没把结构搭稳，就直接把第二阶段逻辑硬塞进去。
- 不允许第二阶段只接一半链路，就宣布功能完成。
- 如果是给旧模块补新能力，也同样按这两个阶段处理：
  - 先补界面结构和入口
  - 再补真实链路与持久化

## 5. 新模块开发指南

适用于新增一个新的模式、页面或独立业务模块。

### 5.1 开工前必须明确

1. 模块名称
2. 模块核心主功能
3. 模块不负责什么
4. 是否需要快捷键
5. 是否需要独立存储
6. 是否需要前后端配置
7. 是否依赖已有共享能力

### 5.2 推荐开发顺序

1. 先写模块文档草稿
2. 先做第一阶段：界面与结构
3. 再做第二阶段：后端、存储、运行时行为
4. 再补快捷键、日志、恢复逻辑
5. 最后再做样式和体验优化

### 5.3 模块最低检查项

- 入口已出现
- 刷新后状态不丢
- 重启后状态不丢
- 默认值完整
- 与旧模块不冲突
- 文档已落地

## 6. 新配置项开发指南

凡是新增开关、下拉、滑块、按钮、文本配置，都按下面检查。

### 6.1 必须同步的位置

1. 前端状态默认值
2. 前端页面渲染与回填
3. 前端保存请求负载
4. 后端状态聚合返回
5. 后端保存逻辑
6. 本地默认值
7. 本地读取逻辑
8. 运行时应用逻辑
9. 模块文档说明

### 6.2 历史高频错误

- 新增了开关，但刷新后恢复默认
- 新增了下拉，但保存后并未真实切换运行时
- 新增了快捷键项，但没有注册处理函数
- 新增了按钮，但只是界面，实际无动作
- 页面内刷新可恢复，但浏览器原生刷新后丢失未落盘改动
- 只验证了自动保存成功路径，没有验证“保存未完成时的草稿恢复路径”
- 新增了字段行，但关键主键未填写，前端校验未通过，结果后端根本没有收到有效数据；再次加载时误以为是“后端没加载”
- 快捷键冲突校验只按按键文本判断，没有带上作用域，结果把不同模块内本来允许重复的按键误判成冲突

以后排查“新增内容再次加载后不见了”时，必须先分清三层：

1. 前端是否真的发出了有效保存数据
2. 后端是否真的写入本地存储
3. 重新加载时是否真的从本地存储读回

不要把“前端未通过校验导致未保存”误判成“后端未加载”。

快捷键验收也必须明确：

1. 同一作用域内不允许重复
2. 不同作用域内允许按键重复时，校验必须按“作用域 + 按键”判断
3. 不能只按按键文本做全局去重

以后同类问题一律先按这一节自查。

## 7. 快捷键开发指南

新增快捷键时，至少检查：

1. 本地初始化定义
2. 处理函数映射
3. 作用域条件
4. 注册与反注册
5. 默认配置
6. 前端兜底定义
7. 后端快捷键定义
8. 界面说明文案

如果快捷键属于某个悬浮窗，必须做独立作用域，不能偷用全局逻辑。

## 8. 提交与归档规范

每次重要提交时，说明至少包含：

1. 这次实现了什么
2. 这次没有实现什么
3. 当前可回滚锚点是什么
4. 下一步从哪里继续最安全

不要只写“修复问题”“更新”这类无信息提交说明。
